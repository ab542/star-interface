# 企业接口自动化实践 - 接口依赖处理方案

## 问题场景

在本项目中，`monitor/list`（监控账户列表）接口的 `mediaIds` 参数需要从 `monitor/unique/list`（监控中心用户筛选条件列表）接口获取真实有效的媒体账户ID。这是典型的**接口依赖**场景。

---

## 企业主流处理方案

| 方案 | 适用场景 | 企业使用频率 | 优点 | 缺点 |
|------|----------|--------------|------|------|
| **1. 测试数据预准备** | 依赖数据在测试环境相对稳定 | ⭐⭐⭐⭐⭐ (最常用) | ✅ 简单可靠，无需复杂框架改造<br>✅ 测试执行速度快<br>✅ 符合数据驱动设计理念 | ❌ 数据变化后需要人工维护 |
| **2. 前置接口调用 + 动态提取** | 数据每次都变化，必须实时获取 | ⭐⭐⭐⭐ | ✅ 全自动，不需要人工维护数据<br>✅ 通用灵活可复用 | ❌ 增加测试执行时间<br>❌ 框架复杂度提升 |
| **3. Pytest Fixture 依赖注入** | 固定依赖关系，整个测试会话只获取一次 | ⭐⭐⭐⭐ | ✅ 复用性好，只调用一次节省时间<br>✅ 利用Pytest原生机制 | ❌ 耦合度高，不够灵活 |
| **4. 测试工厂模式 / Setup 脚本** | 复杂业务链路，需要先创建资源再测试 | ⭐⭐⭐ | ✅ 适合CI/CD流水线，每次全新创建<br>✅ 环境干净不污染 | ❌ 开发成本高 |
| **5. Mock 依赖接口** | 下游接口不稳定或第三方依赖 | ⭐⭐⭐ | ✅ 不依赖外部环境，测试稳定<br>✅ 可控性强 | ❌ Mock不准可能漏问题 |

---

## 针对本项目的建议

### 当前场景分析

- 依赖关系：`monitor/list` 筛选接口 ← `mediaIds` 参数 ← `monitor/unique/list` 筛选条件列表接口
- 数据性质：媒体账户属于业务基础数据，在测试环境相对稳定
- 测试目标：验证筛选功能逻辑正确性，不依赖具体最新数据

### 推荐方案：**测试数据预准备**

**理由：**

1. **简单就是美** - 本项目本身就是数据驱动设计，保持现状不需要改代码
2. **筛选逻辑验证不需要真实最新数据** - 只要验证筛选功能是否正常工作，有几个ID足矣
3. **维护成本低** - 就算ID失效了，手动跑一次 `unique/list` 复制过来就行
4. **不影响测试执行速度**

**操作步骤：**

1. 先运行 `unique/list` 接口测试，从响应中获取几个真实可用的 `id`
2. 把真实 ID 更新到 `monitor_list.yaml` 的 `mediaIds` 位置
3. 运行 `monitor/list` 接口测试验证筛选功能

### 备选方案：**前置接口动态提取**

如果追求全自动，可以扩展YAML语法支持 `pre_steps` 前置步骤，格式示例：

```yaml
- name: "监控中心 - 获取监控账户列表-按媒体账户Ids筛选"
  pre_steps:
    - name: "获取筛选账户列表"
      method: POST
      url: /api/media/monitor/unique/list
      json:
        page: 1
        size: 10
      extract:
        mediaIds: "data.items[0:2].id"  # JSONPath提取前两个item的id
  method: POST
  url: /api/media/monitor/list
  headers:
    Content-Type: application/json
  json:
    page: 1
    size: 10
    mediaIds: "{{pre_steps.0.extract.mediaIds}}"
  expected_status: 200
  expected_response:
    code: 0
  max_retries: 3
```

这种方案需要修改 `tests/star_digi+/test_star_api.py`，增加：
1. `pre_steps` 解析和执行
2. JSONPath 数据提取
3. 变量替换到当前请求
